Repo ~/Downloads/lego/legolization (branch initial-implementation-followup) · evaluated 2026-07-11 against the twelve papers in references/ + papers PDFs/, the vendored StableLego reference implementation found at ~/Downloads/lego/StableLego-main, and a fresh literature sweep.

Verdict: a genuinely solid, honestly documented implementation of milestones M1–M5. The stability core is a constant-for-constant, lever-for-lever port of the StableLego release — and in one respect provably better than its own documentation claims. The placement layer is a well-engineered hybrid of Luo 2015 and Kollsker heuristics, correctly labeled as such in ROADMAP.md, which tracks most known gaps accurately. The audit surfaced no critical correctness bugs, but did find a handful of impactful issues the ROADMAP does not track — chief among them an inert side-press force model, a nearly constant aesthetics metric, a vacuous single-component check, and a quiet placement-quality regression that left the shipped examples stale.

1 · Empirical verification

Everything below was run fresh during this audit, not taken from the docs:

  • Gates: pytest 77 passed (1.7 s); ty and pyrefly clean; ruff format/check clean on src tests. The CLAUDE.md instruction to run ruff ... . repo-wide currently fails only because of the untracked PDF-conversion helper under papers PDFs/.
  • Examples: all three build and report STABLE — pyramid.npy (17 bricks, 59.3 g), arch.npy (30 bricks, 24.9 g), heart.vox (40 bricks, 19.8 g); --strategy luo on heart gives 18 bricks / 21.4 g. Exit code 0 in every case.
  • Determinism: two same-seed CLI runs produce byte-identical output; changing --seed changes the layout. (An apparent nondeterminism was a false alarm — the output filename is embedded in the .ldr header.)
  • Staleness: the shipped data/examples/heart.ldr has 31 bricks; current code produces 40. The post-review commit 66f4f19 (greedy sweep-order change, removal of the redundant bond exp(·0) term) regressed heart's brick count by ~29% and the examples and README output block were never regenerated.
  • Spot-checks reproduced by hand: the two-disconnected-towers layout reports component_count == 1 (finding F4); a stack-bond wall scores 0.833 vs. 0.667 for a stretcher-bond wall on the aesthetics metric (finding F2); the LP and MILP solvers agree on stressed and overloaded cantilever cases (section 3).

3 · Stability core vs. the papers

Bottom line: a high-fidelity port, verified against the reference. The audit compared stability/ line-by-line with the StableLego paper and its released implementation (StableLego-main/py_scripts/stabiliy_analysis.py + config.json, discovered locally):

  • Constants are exact: T = 0.98 N, α = 10−3, β = 10−6, g = 9.8, lever unit 0.0078 m, plate height 0.0032 m — every value matches the release's config.json, and the brick masses in catalog.json match StableLego Table I exactly for every part it lists.
  • Contact geometry is exact: the 4-point diamond (±0.25 pitch) and the 3-point triangle ((0.125,−0.125), (−0.25,0), (0.125,0.125)), and the min-footprint-dimension rule for choosing between them, replicate the released code (which itself simplifies the paper's Q×X interior-knob rule — a divergence ROADMAP already tracks).
  • The objective is term-for-term Eq. (7): L1 equilibrium residuals + α·Σ per-brick max bottom drag + β·Σ all drags; the capacity T is deliberately not a solve-time constraint (the paper says so explicitly) and is applied at scoring per Eq. (8). The score semantics (1 = collapsing, else drag/T) match.
  • Newton's third law by shared variables is an exact reformulation of the release's equality constraints, and the ground model (unpaired reactions that can pull down, StableLego baseplate-style) matches.

Positive finding — the LP relaxation is provably tight, not "slightly optimistic". Because each contact point's normal and drag columns are exact negatives of each other, any solution with both positive can subtract the minimum from both, leaving every equilibrium residual unchanged while strictly reducing the β term. So every LP optimum already satisfies the bilinear non-coexistence constraint that the docstring claims was "dropped" — the convex relaxation is exact, and the --milp mode is redundant except as a cross-check. This was also confirmed empirically: LP and MILP produce identical scores and identical unstable sets on stressed and overloaded cantilever cases. The docstring's attribution is also wrong: BrickGPT §4.2 keeps non-coexistence and solves the nonlinear program with Gurobi (F11).

HighF1 — Side-press forces are torque-inert

The repo models one shared compression variable per brick pair per axis, applied at the mean shared-face centroid. For same-height neighbours that point sits at both bricks' centroid height, so the lever rz is zero — and with the τz row dropped (as in the release), horizontal side forces end up with no torque coefficient at all under gravity loading. The StableLego release gives every horizontal press force a fixed h/2 vertical lever; Luo goes further and puts side-contact normals at the face corner points, which is exactly the mechanism behind his Fig. 7(b) bridge staying stable at 11 segments instead of 5. Structures that transfer weight through side faces will be judged strictly more conservatively here than in either reference. The physics-fidelity backlog does not mention it, and the test suite contains zero side-contact cases, so nothing would catch it.

stability/model.py:198–204 · graph.py:153–178 · StableLego release stabiliy_analysis.py:174–186 · Luo §6

Minor, disclosed deviations (all match the StableLego release where they diverge from the paper text): two torque axes instead of three; per-brick score reads only bottom-interface drags (localization shifts one brick down vs. the paper's Eq. 6/8, verdict unaffected); the 3-point triangle stays axis-aligned under yaw. The hollow shell being 1 plate thick vs. Luo's ~3 voxel-widths (F10) is compensated by the stability-aware restore loop — a mechanism neither paper has, which is sound (monotone material addition, bounded rounds, honest failure reporting) but heuristic: trouble columns derive from the unstable bricks' own cells and may miss a root cause elsewhere in the support path.

4 · Placement & layout vs. the papers

Bottom line: a structured, honest hybrid — not a faithful Luo, and mostly says so. The two-phase split-and-remerge skeleton, the strict acceptance discipline, and the k = ⌊f/10⌋+1 ring growth are genuinely Luo's (Algorithms 4, 5, 7; Eq. 1). The maximal random merge is a faithful Algorithm 4 (random pair pop, liveness checks, union-in-catalog rule). What differs: the atoms (1×1 plates, not bricks — a deliberate generalization to plate resolution), the stability metric (StableLego RBE instead of Luo's maximin QP — arguably an upgrade, but a different ordering, F6), the critical-portion selection (all border/unstable bricks instead of one importance-sampled brick — larger, more disruptive regions), the colour model (hard uniformity only, F7), and failMAX (20/30 vs. Luo's 100 — disclosed in the docstring).

HighF3 — Why repairs inflate brick counts ~3×

Repair splits regions to 1×1 plates so columns can re-phase — but merged_rect requires equal heights, so 2D remerge can only ever make bigger plates (there is no 2-plate-tall part to pass through), and the only path back to bricks is compact_vertical, which demands three stacked plates with identical footprints and colour. After a random remerge, adjacent plate layers almost never align, so repaired regions permanently carry ~3× the part count. ROADMAP tracks the symptom (hollow sphere ~374 vs. ~200 bricks); this is the mechanism. Compounding: component_border seeds whole seam planes at once, and greedy never runs a global re-merge pass after placement.

placement/merge.py:122–148, 232–303 · greedy.py:62–64 · ROADMAP "Cheaper repairs"

HighF2 — The aesthetics metric barely discriminates

_seam_alignment counts seams per plate-layer cell, so the three stacked seam cells inside every brick-to-brick vertical interface make ⅔ of all seams "repeated" no matter the bond pattern. Measured: a perfect stretcher-bond wall scores 0.667, a worst-case stack-bond wall 0.833 — a usable range of ~0.17 where the objective weight assumes [0, 1]. The fix is counting per brick-pair interface (top seam plate vs. the seam plate above), not per cell.

placement/base.py:101–118 · verified empirically in this audit

MediumF5 — Reinforcement's "random rebuild" isn't random

The greedy docstring promises delete-and-rebuild "with a different random order", but _fill iterates seeds in sorted (z, x, y) order; the only randomness is a 10−3 jitter that breaks exact score ties. Rebuild attempts therefore tend to regenerate the same local layout while the failure counter climbs — each failed attempt costing a full LP evaluation — and the loop's effective search is far weaker than both the docstring and Luo's design intend. The code half-admits this: connectivity failures are delegated to improve_connectivity precisely because "the largest-first fill would just recreate them".

placement/greedy.py:7–8, 76, 84, 127–168

Also verified on the positive side: slope geometry and rotation bookkeeping were hand-checked for all four yaws against the real LDraw part files (3040b.dat, 3039.dat, 4286.dat) — origins, descent direction, and the origin_offset math all reproduce exactly; slope and tile swaps are collision- and connectivity-safe by construction; the yaw-90 anchoring convention (place_rect) is correct; catalog LDraw part numbers are all genuine. Kollsker-side gaps beyond F9: no perpendicularity reward (Bao/SM-GA's crossing-direction term appears nowhere in scoring), no random layer build directions, no per-layer incomplete-construction repair, and lexicographic (size, bond) candidate ranking rather than an additive cost — the latter tracked in ROADMAP.

5 · Pipeline, I/O, CLI

Bottom line: ordering is sound; the transforms are right; the edges are soft.

  • Pipeline order (hollow → place+refine → stability-aware restore ×5 → opt-in slopes/tiles → re-verify → export) matches Luo's staging; the restore loop is an extension beyond the papers, not an ordering bug. The ROADMAP claim that stability is re-verified after finishing is true — though it is verification only, with no repair loop if a slope/tile pass destabilizes the model.
  • The .vox parser is correct where it counts: MAIN-chunk descent, XYZI layout, the 1-based palette shift (vox_palette[1:] = raw), z-up direct mapping. The composite voxel→LDraw map has determinant +1 — no mirroring, the classic pitfall avoided. Robustness gaps are F14.
  • The LDU transform is verified: X = 20x, Z = 20y, Y = −8·(layer + height) against real part files; one 0 STEP per plate layer sorted by bottom layer means every emitted prefix is physically assemblable stud-up; .mpd gets a correct FILE/NOFILE wrapper.
  • CLI: flags match the README; exit codes 0/1/2 as documented; the buildability triple (stable ∧ single-component ∧ grounded) is enforced — but see F4: the component check is redundant with the floating check because the ground node merges all grounded islands, so "single-component" is weaker than Luo's brick-graph single-connectedness and weaker than the README's plain-language claim. Two disconnected towers on the ground exit 0.
  • Colour: redmean quantization is correctly implemented and a reasonable middle ground (better than raw RGB distance, short of CIELAB — and notably, no reference paper actually specifies better; the lambrecht paper has no colour treatment at all). Name-token filtering of non-solid finishes is the right workaround for pyldraw3's unreliable colour metadata.

7 · Test-suite assessment

Strengths. The suite (72 functions / 77 cases) is unusually substantive for a project this age: the stability tests are analytic rather than smoke (floating-brick collapse, tile-top collapse, cantilever stress monotonicity, overload past T, weakest-pair identity), the placement tests pin the exact-cover/no-overlap/colour invariants for both strategies plus random grids, and the LDraw tests pin positions including the slope origin. The transform and geometry layers are well covered.

Gaps, in rough priority order:

  1. No quantitative physics pins. The single-stud cantilever has a closed-form optimum (drag_max = 1.25·W ≈ 0.019233 N; score 0.019625) that would pin contact geometry, lever units, and objective in one assertion; currently only > 1e-4 is asserted, so a large uniform constant error could pass every test.
  2. The StableLego release's test_lego/*.json fixtures sit unused on this machine — the exact cross-validation ROADMAP calls "the best available ground truth".
  3. Zero side-contact tests, which is why F1 is invisible to the suite. No 3-pt vs 4-pt verdict test, no ground-uplift test, no torque-sign/symmetry test.
  4. The hollow-restore loop's reason for existing — hollowing breaks stability, restore fixes it — is never exercised; test_hollow.py is purely geometric and the pipeline test uses a shape already stable when hollowed.
  5. LuoStrategy's refinement loops are likely never entered (its only fixture, a 6×6×3 pyramid, merges into a stable connected layout), so the acceptance logic and _better have effectively no coverage. improve_connectivity has no direct test.
  6. No determinism test and no brick-count regression metric — either would have caught the heart 31→40 regression (F12) at commit time.
  7. LP-vs-MILP agreement is tested only on a trivially stable 2-brick stack; the interesting (stressed/unstable) cases were verified manually in this audit but should be pinned.

8 · Documentation accuracy

  • ROADMAP.md is the standout: nearly every deliberate simplification found in this audit that the authors knew about is disclosed there, accurately — including the exact hollow-sphere brick numbers and the honest "slopes add material, hence opt-in" framing. The "72 tests" claim is exact at the function level.
  • README oversells in three places: "covers every voxel with real parts — bricks, plates, tiles, and 45°/33° slopes" (placement uses bricks+plates only; tiles/slopes are opt-in post-passes; the 33° slope is never placed by any code path); the sample output block shows the stale 31-brick heart; "single-component" means the weaker ground-merged sense (F4).
  • solver.py's self-description is wrong in the repo's favour (F11): the LP is exact, not "slightly optimistic", and it is not what BrickGPT does.
  • pyproject.toml drift: unused runtime deps (pyperclip, textual), a dead SCIP fallback (no pyscipopt), a docs URL to a nonexistent GUIDE.md.

10 · Prioritized recommendations

Ordered by leverage per unit effort; none applied yet, per the audit's no-code-changes scope.

  1. Give side presses the reference lever (h/2 like the StableLego release, or corner points like Luo) and add the missing side-contact tests. Small, isolated change in build_model; unlocks Luo-style side-supported structures. (F1)
  2. Cross-validate the solver against the local StableLego-main/test_lego/*.json fixtures and pin the analytic cantilever number 0.019233 N as a golden test. This is the cheapest possible hardening of the core everything else trusts, and the fixtures are already on disk. (§7)
  3. Fix _seam_alignment to count per brick-pair interface; add the two-wall baseline test that would have caught it. (F2)
  4. Make reinforcement rebuilds actually random (shuffle seed order per attempt, or honestly re-document), and regenerate the stale examples + README block; add a brick-count regression assertion so the next quality regression is visible in CI. (F5, F12)
  5. Decide the component semantics: either check brick-graph single-connectedness excluding ground (Luo's definition, README's implication) or document the baseplate-merged meaning and drop the redundant check. (F4)
  6. Correct the solver docstring (exact, not optimistic; not BrickGPT's method) and consider demoting --milp to a debug cross-check. (F11, F15)
  7. Weigh plates realistically (BrickLink masses) — one JSON edit that removes a systematic optimistic bias exactly where repaired layouts are plate-heavy. (F8)
  8. Colour-blind interiors: implement Luo's IGNORE label for hollow-restored fill so invisible cells stop fragmenting merges. Pairs naturally with the tracked "cheaper repairs" work. (F7, F3)
  9. Dependency and robustness sweep: drop pyperclip, gate textual, add pyscipopt or remove SCIP, wrap .vox struct errors, validate integer .npy codes, wire --no-refine through to luo or reject the combination. (F13, F14)
